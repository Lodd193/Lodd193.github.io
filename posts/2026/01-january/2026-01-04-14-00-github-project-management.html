Managing Projects with GitHub: Lessons from Building a Workout Tracker

GitHub has a lot of 'hidden' features, it's way more than what I first thought it was - a place to stash my code. While building my workout tracker, I looked into these features a lot and discovered DevOps! I quickly learned how powerful GitHub's project management and DevOps features are. Here's what worked, what I made a huge mess of, and what I'd do differently next time.

Why GitHub for Project Management?

When I started the workout tracker project, I knew I needed version control - I constantly made small iterative changes/improvements based on my own feedback from testing my app at the gym. What I didn't realise was how much GitHub could help me stay organised, track progress, and maintain the quality of my code throughout development.

Coming from a business background with my MBA, I am very familiar with project management principles, but applying them to software development was new to me GitHub's tools really helped to bridge that gap.

Version Control: The Foundation

Commits as Documentation

Early on, I made the mistake of writing vague commit messages like "fixed stuff" or "updates". This came back to haunt me when I needed to track down when a bug was introduced - similar to my early Python days when I didn't comment on my code and when I created monolithic blocks of code which I couldn't understand when I revisited them weeks or months later.

I learned to write super descriptive commit messages that explain the "why", not just the "what":

  • Before: "update auth"
  • After: "Add email confirmation to prevent spam signups"

This new found discipline made it a lot easier to understand my project's history and debug issues weeks later.

Branching Strategy

I adopted a simple branching strategy:

  • main - Production-ready code only
  • dev - Integration branch for features
  • feature/* - Individual features (e.g., feature/workout-templates)
  • fix/* - Bug fixes (e.g., fix/pr-calculation)

This kept my main branch stable and allowed me to experiment with new features without breaking my working code (I used to break my working code constantly - honestly, version control is a life saver and something I wish I had learnt on day 1 of my data science journey). When a feature was ready, I'd merge it to dev for testing, then to main for deployment.

Pull Requests: Code Review for Solo Developers

Even though I was working solo, I still used pull requests. This might seem excessive, but it helped me learn and feel comfortable using these commands and it forced me to:

  1. Review my own code with fresh eyes before merging
  2. Write a summary of what I'd changed and why
  3. Check that tests still passed
  4. Catch bugs I'd missed during development

The act of writing a PR description made me think critically about whether my implementation was actually solving the problem correctly (the MBA came into play here).

Issues: Tracking Features and Bugs

I used GitHub Issues as my personal backlog:

  • Feature requests: "Add workout templates for common routines"
  • Bugs: "Personal records not calculating for cardio exercises"
  • Enhancements: "Improve mobile performance on exercise selection"

I labelled issues by type (bug, enhancement, feature) and priority (high, medium, low). This helped me focus on what mattered most and avoid scope creep.

When I fixed an issue, I'd reference it in my commit message (Fixes #12), which automatically closed the issue and created a link between the commit and the problem it solved.

GitHub Projects: Visual Progress Tracking

I created a GitHub Project board with three columns:

  • To Do: Features and fixes I planned to work on
  • In Progress: What I was actively developing
  • Done: Completed and merged work

Dragging cards across the board gave me a visual sense of progress and helped me avoid taking on too many tasks at once, something I used to do a lot. It also made it easy to see what I'd accomplished over time, which helped me stay motivated.

GitHub Actions: Automated Workflows

This is where things got really interesting. I began learning about GitHub Actions and using it to automate repetitive tasks:

Continuous Integration

I created a workflow that ran on every push to check:

  • TypeScript compilation (no type errors)
  • Linting (code style consistency)
  • Tests (all passing)

This caught errors before they reached production and gave me confidence that my code was solid - I can't begin to tell you how much of a game-changer this is! Everyone who is setting out on their coding journey should learn this asap - it'll change your life.

Automated Deployments

I also connected my GitHub repository to Vercel for automatic deployments. Every commit to main triggered a production deploy, and every PR created a preview deployment with a unique URL.

This was game-changing - I could test features in a production-like environment before merging, and I always knew the live site matched what was in my main branch.

What I Learned

Git Tidyness Matters

Small, focused commits are easier to understand and revert if needed. Large commits that change 20 files across multiple features are a pain in the @$$ to debug later - just don't do it, work iteratively, it's what GitHub excels at!

Documentation Through Process

Good commit messages, PR descriptions, and issue tracking created automatic documentation of my decision-making process. When I returned to the project after a break, I could quickly get back up to speed, and, crucially, if I were to collaborate with anyone, they could pick up where I left off. Imagine working with someone in a different timezone, you go to bed, and they wake up. The project you're working on starts seeing progress over a much longer window - the possibilities are endless.

Automation Saves Time

Setting up GitHub Actions took a few hours and mainly consisted of watching YouTube videos ("Techworld with Nana" was super helpful). Still, it saved me countless hours of manual testing and deployment. The initial investment paid off right away

Solo Doesn't Mean Sloppy

Using professional workflows (branches, PRs, code review), even as a solo developer, prepared me for collaborative work and kept my codebase maintainable.

What I'd Do Differently

  • Start with issues earlier: I wrote a lot of code before creating issues to track it. Starting with issues would have helped me plan better.
  • Use templates: GitHub lets you create templates for issues and PRs. This would have saved time and ensured consistency.
  • Set up branch protection: I could have required PR reviews (even self-reviews) and passing tests before merging to main.
  • Tag releases: I should have used Git tags to mark significant versions (v1.0, v1.1, etc.) for easier rollback if needed.

Tools I Used

  • Git CLI: For commits, branches, and merges
  • GitHub Web UI: For PRs, issues, and project boards
  • GitHub Desktop: Occasionally, for visualising complex merge conflicts
  • VS Code Git integration: For quick staging and committing during development

Key Takeaways

  1. Version control is more than backup: It's a complete project management system when used properly.
  2. Process scales: Good habits on solo projects translate directly to team environments.
  3. Automation is powerful: GitHub Actions can handle testing, deployment, and more - freeing you to focus on building.
  4. Documentation happens naturally: When you commit, review, and track issues properly, you're creating project documentation without extra effort.

GitHub transformed from "just a code host" into my complete DevOps platform. Whether you're building a personal project or working on a team, these workflows will keep you organised and productive.

If you're new to GitHub project management, start simple: write better commit messages, create a few issues, and try using a project board. You'll be surprised how much clearer your development process becomes.

Building A Full-Stack Workout Tracker

04 January 2026 ‚Ä¢ 8 min read ‚Ä¢ üî® Building

I built a comprehensive workout tracking Progressive Web App using Next.js, React, TypeScript, and Supabase. Here's how I went from concept to a fully-deployed fitness companion with offline support, analytics, and 70+ exercises.

Context

Like many people trying to stay consistent with fitness, I've struggled to track my workouts effectively to ensure I'm hitting my cardio goals and keeping up with progressive overload in my weightlifting. I'd used various apps before, but they were either too complicated, lacked the features I wanted, or required subscriptions for basic functionality. So I wanted something tailored to my own needs that could:

  • Log workouts quickly without friction
  • Track progress over time and provide useful visualisations
  • Work offline for those holiday hotel gyms where signal is poor
  • Show my personal records and help me to track achievements
  • Provide templates for common workout routines

More importantly, I wanted to build something real that combined my skills and showcased frontend development, backend integration, authentication, and data visualisation - all skills that I had developed through my MSc and bootcamp training but needed more real-world/hands-on experience with as that is where the bugs and issues are encountered and the real learning begins.

The Problem

Building a full-stack application from scratch presented challenges I hadn't tackled before:

  • State management complexity: Managing workout sessions, exercise data, user preferences, and real-time updates across multiple components
  • Offline functionality: Ensuring the app worked without internet and synced properly when reconnected
  • Authentication & security: Implementing proper user authentication while ensuring data privacy (users should only see their own workouts)
  • Data modelling: Designing a database schema that could handle flexible workout tracking (different exercises, varying sets/reps, cardio vs strength)
  • Progressive Web App setup: Making it installable and work like a native app across devices

I also needed to decide on a tech stack that would be modern and - something I could actually deploy myself.

Investigation & Planning

Technology Choices

After researching options, I settled on a stack that balanced power with experience:

  • Next.js 16 & React 19: Server-side rendering, excellent developer experience, and built-in API routes
  • TypeScript: Type safety to catch errors early and improve code quality (My TS skills are nowhere near my Python skills currently, this was a steep learning curve)
  • Tailwind CSS: Rapid UI development with utility classes
  • Supabase: PostgreSQL database with built-in authentication, real-time subscriptions, and Row Level Security (RLS)
  • Vercel: Easy deployment with automatic HTTPS and global CDN

Supabase was a game-changer because it gave me a PostgreSQL database, authentication, and real-time capabilities without requiring me to set up my own backend infrastructure. This likely saved tens of hours of work.

Feature Planning

I broke down the MVP into core features:

  1. User authentication (sign up, login, password reset)
  2. Exercise library (70+ exercises across 14 categories)
  3. Workout logging (sets, reps, weight)
  4. Progress tracking (personal records, workout history)
  5. Basic analytics (calendar heat map, streak tracking)
  6. PWA capabilities (offline support, installable)
  7. Repeatability & Progression - ensuring it was super simple to repeat a workout template and receive tailored feedback on what to do to improve on the previous session

Implementation

Database Design

I designed the Supabase database schema with three main tables:

  • users - User profiles linked to Supabase Auth
  • workouts - Individual workout sessions with date and user reference
  • exercises - Exercise entries within workouts (sets, reps, weight)

The key insight was to use Row Level Security (RLS) policies to ensure users could access only their own data. This meant I didn't need complex backend permissions logic‚Äîthe database enforced isolation automatically.

Authentication Flow

Supabase Auth handled the heavy lifting, but I needed to:

  1. Create an authentication context to manage user state across the app
  2. Implement protected routes with middleware to redirect unauthenticated users
  3. Handle session persistence and automatic token refresh
  4. Add email confirmation for new signups

The middleware pattern in Next.js 16 made route protection elegant‚Äîany route under /dashboard automatically requires authentication.

Workout Logging Interface

The logging interface needed to be fast and intuitive. I built it with:

  • Category selection: Users pick a muscle group (Chest, Back, Legs, etc.) or cardio type
  • Exercise selection: Filtered list of relevant exercises
  • Set entry: Quick add/remove sets with reps and weight tracking
  • Personal record indicators: üèÜ badges automatically appear when a new PR is achieved

Real-time PR detection was particularly satisfying - I calculate 1RM estimates using the Epley formula and compare against historical data as users log sets.

Analytics & Visualisations

I implemented several analytics features to keep users motivated:

  • Calendar heat map: GitHub-style grid showing workout consistency over time
  • Streak tracking: Current and longest workout streaks
  • Weekly cardio progress: Visual progress toward the 150-minute weekly goal
  • Exercise history: Per-exercise progress charts showing weight/reps over time

The heat map was the most complex‚ÄîI had to aggregate workout data by date, handle time zone differences, and render a dynamic grid that responds to the current year and month.

Progressive Web App Setup

Making it a PWA involved:

  1. Creating a service worker for offline caching
  2. Adding a web app manifest with icons and theme colours
  3. Implementing offline detection and sync queue
  4. Optimising for mobile performance

The offline functionality was crucial as workouts logged without internet are queued locally and automatically sync when connectivity returns. This required state management to handle the pending uploads and conflict resolution.

Templates System

To speed up logging, I added workout templates:

  • Pre-made routines: PPL (Push/Pull/Legs), Upper/Lower, Full Body, Bro Split, Starting Strength
  • Custom templates: Users can save their own favourite workout combinations
  • One-click loading: Templates populate the logging interface instantly

This was a huge time-saver for recurring workouts - no more manually selecting the same exercises every session and I ensured these were saved on SupaBase rather than Local Storage to ensure they would be available on any device.

Challenges & Solutions

Challenge 1: TypeScript Complexity

TypeScript was new to me, and I initially struggled with complex type definitions, especially around Supabase query responses and React component props.

Solution: I learned to use TypeScript's utility types (Partial, Pick, Omit) and created shared interface definitions in a types directory. The investment paid off, and type errors caught bugs before runtime. Coming from Python, this was a steep learning curve.

Challenge 2: Real-Time Updates

Getting Supabase real-time subscriptions to work reliably was tricky, especially managing subscription cleanup to prevent memory leaks.

Solution: I used React's useEffect cleanup functions properly and ensured subscriptions were tied to the component lifecycle. I also added manual refresh buttons as a fallback.

Challenge 3: Mobile Performance

Initial load times on mobile were slower than I wanted, especially with the exercise library.

Solution: I implemented code splitting, lazy-loaded heavy components, and moved the exercise library to a static JSON file rather than querying the database every time. This reduced the initial bundle size significantly.

What I Learned

Technical Skills

  • Full-stack development: Connecting frontend UI to backend database with authentication
  • TypeScript proficiency: Writing type-safe code and understanding generics
  • Database design: Normalization, relationships, and security policies
  • PWA implementation: Service workers, caching strategies, and offline-first design
  • State management: Using React Context effectively for global state
  • Deployment: Environment variables, CI/CD with Vercel, and production monitoring

Product & UX Insights

  • Friction matters: Every extra tap or screen reduces usage. The logging flow needed to be seriously optimised.
  • Visual feedback is motivating: The PR badges and streak tracking were fulfilling.
  • Offline support is non-negotiable: An app that doesn't work offline would be useless in this context.
  • Templates save time: Users don't want to think about what to log - they want to start lifting and know the sets/reps/weight to use for progressive overload.

Workflow Improvements

This was my first project using a component-driven dev approach. Breaking the UI into small, reusable components made debugging easier and sped up feature development. I also learned the value of thoroughly testing authentication flows - auth bugs are a nightmare and sapped many hours of my time.

  • ‚úÖ Installs as a PWA on mobile and desktop
  • ‚úÖ Tracks 70+ exercises across 14 categories

More importantly, I use it myself every workout. It's proven that I can take a project from idea to production deployment, which was a major confidence boost as everyone in the tech world seems to be a super-expert with many years of experience doing everything, or at least that's how I perceived it.

Next Steps

While the core app is complete, there are features I'd like to add:

  • Exercise videos/GIFs: Form demonstrations for each exercise
  • Workout notes: Add text notes to sessions (energy levels, injuries, etc.)
  • Data export: Download workout history as CSV
  • Social features: Optional workout sharing or challenge friends
  • Advanced analytics: Volume load over time, muscle group balance, recovery patterns
  • Mobile app: Native iOS/Android apps using React Native

The codebase is also a useful foundation for future full-stack projects. I've learned patterns I can reuse, and I'm much more confident with TypeScript, Next.js, and Supabase (I didn't even know Supabase existed before this project!)

Key Takeaways

  1. Start with MVP: I could have added dozens more features initially, but creating something usable first was the right call - iterative improvements are the way to go. Map out your ideas - the 1) What, 2) Why, 3) How, 4) When, before you write any code. An project plan template is free to use on the Resources page
  2. TypeScript is worth it: The learning curve was steep, but type safety caught so many bugs, and it's a very in-demand skill right now.
  3. Supabase accelerates development: Not having to build authentication and database infrastructure saved weeks.
  4. Real projects teach more than tutorials: Hitting unexpected problems and solving them built real skills.
  5. Deploy early, iterate often: Getting it live - even imperfect - forced me to polish rough edges.

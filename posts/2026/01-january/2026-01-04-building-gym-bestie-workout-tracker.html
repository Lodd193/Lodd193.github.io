Building Gym Bestie: A Full-Stack Workout Tracker

04 January 2026 ‚Ä¢ 8 min read ‚Ä¢ üî® Building

I built a comprehensive workout tracking Progressive Web App using Next.js, React, TypeScript, and Supabase. Here's how I went from concept to a fully-deployed fitness companion with offline support, analytics, and 70+ exercises.

Context

Like many people trying to stay consistent with fitness, I struggled to track my workouts effectively. I'd used various apps before, but they were either too complicated, lacked the features I wanted, or required premium subscriptions for basic functionality. I wanted something tailored to my needs that could:

  • Log workouts quickly without friction
  • Track progress over time with meaningful visualisations
  • Work offline at the gym where signal is poor
  • Show personal records and celebrate achievements
  • Provide templates for common workout routines

More importantly, I wanted to build something real that combined frontend development, backend integration, authentication, and data visualisation‚Äîall skills I was developing through my MSc and bootcamp training.

The Problem

Building a full-stack application from scratch presented several challenges I hadn't tackled before:

  • State management complexity: Managing workout sessions, exercise data, user preferences, and real-time updates across multiple components
  • Offline functionality: Ensuring the app worked without internet and synced properly when reconnected
  • Authentication & security: Implementing proper user authentication while ensuring data privacy (users should only see their own workouts)
  • Data modelling: Designing a database schema that could handle flexible workout tracking (different exercises, varying sets/reps, cardio vs strength)
  • Progressive Web App setup: Making it installable and work like a native app across devices

I also needed to decide on a tech stack that would be modern, scalable, and‚Äîcritically‚Äîsomething I could actually deploy without managing servers myself.

Investigation & Planning

Technology Choices

After researching various options, I settled on a stack that balanced power with developer experience:

  • Next.js 16 & React 19: Server-side rendering, excellent developer experience, and built-in API routes
  • TypeScript: Type safety to catch errors early and improve code quality
  • Tailwind CSS: Rapid UI development with utility classes
  • Supabase: PostgreSQL database with built-in authentication, real-time subscriptions, and Row Level Security (RLS)
  • Vercel: Seamless deployment with automatic HTTPS and global CDN

Supabase was a game-changer‚Äîit gave me a PostgreSQL database, authentication, and real-time capabilities without needing to set up my own backend infrastructure.

Feature Planning

I broke down the MVP into core features:

  1. User authentication (sign up, login, password reset)
  2. Exercise library (70+ exercises across 14 categories)
  3. Workout logging (sets, reps, weight)
  4. Progress tracking (personal records, workout history)
  5. Basic analytics (calendar heat map, streak tracking)
  6. PWA capabilities (offline support, installable)

Implementation

Database Design

I designed the Supabase database schema with three main tables:

  • users - User profiles linked to Supabase Auth
  • workouts - Individual workout sessions with date and user reference
  • exercises - Exercise entries within workouts (sets, reps, weight)

The key insight was using Row Level Security (RLS) policies to ensure users could only access their own data. This meant I didn't need complex backend permissions logic‚Äîthe database enforced isolation automatically.

Authentication Flow

Supabase Auth handled the heavy lifting, but I needed to:

  1. Create an authentication context to manage user state across the app
  2. Implement protected routes with middleware to redirect unauthenticated users
  3. Handle session persistence and automatic token refresh
  4. Add email confirmation for new signups

The middleware pattern in Next.js 16 made route protection elegant‚Äîany route under /dashboard automatically requires authentication.

Workout Logging Interface

The logging interface needed to be fast and intuitive. I built it with:

  • Category selection: Users pick a muscle group (Chest, Back, Legs, etc.) or cardio type
  • Exercise selection: Filtered list of relevant exercises
  • Set entry: Quick add/remove sets with reps and weight tracking
  • Personal record indicators: üèÜ badges automatically appear when a new PR is achieved

Real-time PR detection was particularly satisfying‚ÄîI calculate 1RM estimates using the Epley formula and compare against historical data as users log sets.

Analytics & Visualisations

I implemented several analytics features to keep users motivated:

  • Calendar heat map: GitHub-style grid showing workout consistency over time
  • Streak tracking: Current and longest workout streaks
  • Weekly cardio progress: Visual progress toward the 150-minute weekly goal
  • Exercise history: Per-exercise progress charts showing weight/reps over time

The heat map was the most complex‚ÄîI had to aggregate workout data by date, handle different time zones, and render a dynamic grid that responds to the current year and month.

Progressive Web App Setup

Making it a PWA involved:

  1. Creating a service worker for offline caching
  2. Adding a web app manifest with icons and theme colours
  3. Implementing offline detection and sync queue
  4. Optimising for mobile performance

The offline functionality was crucial‚Äîworkouts logged without internet are queued locally and automatically sync when connectivity returns. This required careful state management to handle pending uploads and conflict resolution.

Templates System

To speed up logging, I added workout templates:

  • Pre-made routines: PPL (Push/Pull/Legs), Upper/Lower, Full Body, Bro Split, Starting Strength
  • Custom templates: Users can save their own favourite workout combinations
  • One-click loading: Templates populate the logging interface instantly

This was a huge time-saver for recurring workouts‚Äîno more manually selecting the same exercises every session.

Challenges & Solutions

Challenge 1: TypeScript Complexity

TypeScript was new to me, and I initially struggled with complex type definitions, especially around Supabase query responses and React component props.

Solution: I learned to use TypeScript's utility types (Partial, Pick, Omit) and created shared interface definitions in a types directory. The investment paid off‚Äîtype errors caught bugs before runtime.

Challenge 2: Real-Time Updates

Getting Supabase real-time subscriptions to work reliably was tricky, especially managing subscription cleanup to prevent memory leaks.

Solution: I used React's useEffect cleanup functions properly and ensured subscriptions were tied to component lifecycle. I also added manual refresh buttons as a fallback.

Challenge 3: Mobile Performance

Initial load times on mobile were slower than I wanted, especially with the exercise library.

Solution: I implemented code splitting, lazy-loaded heavy components, and moved the exercise library to a static JSON file rather than querying the database every time. This reduced initial bundle size significantly.

What I Learned

Technical Skills

  • Full-stack development: Connecting frontend UI to backend database with authentication
  • TypeScript proficiency: Writing type-safe code and understanding generics
  • Database design: Normalization, relationships, and security policies
  • PWA implementation: Service workers, caching strategies, and offline-first design
  • State management: Using React Context effectively for global state
  • Deployment: Environment variables, CI/CD with Vercel, and production monitoring

Product & UX Insights

  • Friction matters: Every extra tap or screen reduces usage. The logging flow needed to be ruthlessly optimised.
  • Visual feedback is motivating: The PR badges and streak tracking created genuine excitement.
  • Offline support is non-negotiable: Gyms have terrible signal. An app that doesn't work offline is useless in this context.
  • Templates save time: Users don't want to think about what to log‚Äîthey want to start lifting.

Workflow Improvements

This was my first project using a component-driven development approach. Breaking the UI into small, reusable components made debugging easier and sped up feature development. I also learned the value of testing authentication flows thoroughly‚Äîauth bugs are the worst kind of bugs.

Results

Gym Bestie is now live and deployed. The app successfully:

  • ‚úÖ Works offline with automatic sync
  • ‚úÖ Installs as a PWA on mobile and desktop
  • ‚úÖ Tracks 70+ exercises across 14 categories
  • ‚úÖ Calculates personal records in real-time
  • ‚úÖ Visualises progress with heat maps and charts
  • ‚úÖ Secures user data with RLS and authentication

More importantly, I use it myself every workout. It's proven that I can take a project from idea to production deployment, which was a major confidence boost.

Next Steps

While the core app is complete, there are features I'd like to add:

  • Exercise videos/GIFs: Form demonstrations for each exercise
  • Workout notes: Add text notes to sessions (energy levels, injuries, etc.)
  • Data export: Download workout history as CSV
  • Social features: Optional workout sharing or challenge friends
  • Advanced analytics: Volume load over time, muscle group balance, recovery patterns
  • Mobile app: Native iOS/Android apps using React Native

The codebase is also a great foundation for future full-stack projects. I've learned patterns I can reuse, and I'm much more confident with TypeScript, Next.js, and Supabase.

Key Takeaways

  1. Start with MVP: I could have added dozens more features initially, but shipping something usable first was the right call.
  2. TypeScript is worth it: The learning curve was steep, but type safety caught so many bugs.
  3. Supabase accelerates development: Not having to build authentication and database infrastructure saved weeks.
  4. Real projects teach more than tutorials: Hitting unexpected problems and solving them built real skills.
  5. Deploy early, iterate often: Getting it live‚Äîeven imperfect‚Äîforced me to polish rough edges.

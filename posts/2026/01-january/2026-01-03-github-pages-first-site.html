My first GitHub Pages site

03 Jan 2026 · 4 min read · Shipping

I wanted to set up a simple option to document my tech journey - focussing on developing my SQL and Python DS/ML skills, and learning TS, Node.js & React. GitHub Pages was the best option because it didn't require setup on another platform and made it really easy to share my project updates.

The context

I've been learning across a few areas at once, and I needed somewhere to share my progress properly. I didn't want to start by sharing polished tutorials (there are loads of those out there) and I didn't want to share a stream of half-thoughts. Instead, I opted to focus on short write-ups that I can come back to later.

The problem

The confusing bit wasn't "how to make a website" as I'm already fairly confident with basic HTML and CSS. It was learning more about GitHub itself as I'm still fairly new to the platform. Repositories, commits, branches, and managing all of this from my terminal was a learning curve.

The investigation

  • I checked what GitHub Pages actually serves by default.
  • I looked at where the Pages settings point to: branch and folder.
  • I verified the files existed in the right place and were named correctly.
  • I refreshed a few times and learnt the hard truth about browser caching.

The fix

The fix was simple once I stopped guessing and followed the flow:

  1. Create index.html in the repo root.
  2. Enable Pages in Settings → Pages and deploy from the main branch root.
  3. Add a stylesheet and link it properly.

The key line that made styling work was:

<link rel="stylesheet" href="styles.css" />

What I learned

  • GitHub Pages is predictable. It only feels mysterious when you don't know what it expects - practice makes perfect!.
  • File names and locations matter. A missing index.html is enough to make everything look broken.
  • Commits are basically "save points". Commit equals deploy.
  • If a change doesn't show, it's usually caching, or the Pages build is still catching up.

Next time

  • I'll add pages deliberately instead of linking to pages that don't exist.
  • I'll keep the structure consistent so writing stays easy.
  • I'll treat the site as a system, not a one-off task.

Next step

I'm going to publish my next post using the same structure, then improve the Posts page so it lists posts cleanly as the site grows.
